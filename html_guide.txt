<core_technical_requirements>
## Front-end Style Requirements
- Always use `min-height:[n]px` instead of `height:[n]px` in CSS containers to prevent overflow (use `min-h-[n]px` in TailwindCSS)
- Use horizontal layout for optimal display. Always limit the height of every element
- Leverage Tailwind's responsive design utilities (use Tailwind CSS classes, not inline styles when possible)
- Always give a fixed pixel height or width to limit image elements instead of percentage or flex layout
- The background property in CSS of outside container must be the same in all content slides except the title slide and closing slide
- Treat slide-container as the outer container, do not write any css code outside of it and do not use padding property on it.
- DO NOT use padding bottom property anywhere, only use padding top to maintain the vertical spacing.

## Image Usage Rules
- DO NOT use image generation tools to generate the image unless user specifies the image generation request
- DO NOT WRITE A NON-EXISTENT IMAGE LINK INTO THE HTML FILE. NEVER USE IMAGE WHICH IS NON-LOCAL
- DO NOT use images generated from .mmd (Mermaid), Python scripts, or similar sources, as they can lower slide quality
- DO NOT write inline SVG code in HTML - use actual image files with `<img>` tags instead
- Always use absolute path for images, fonts, and other resources, for example: `<img src="/home/ubuntu/project/image.png" />`
- Strictly follow the image plan provided and use ONLY the image path(s) specified for the slide (if any) to prevent duplicate images
- All non-background images MUST use 'object-fit: contain' CSS property or Tailwind's 'object-contain' utility class
</core_technical_requirements>

<design_guidelines>
## Visual Design Rules
- DO NOT use website UI styles like rounded corners, card components with border accents (border-left/right), and shadowed boxes
- DO NOT use CSS animations, @keyframes, transitions, hover effects - slides are static presentations, not interactive websites
- Minimize card style usage - it makes pages fragmented. Never use card style in data visualization pages
- Vary the visual structure when appropriate (except title slide and closing slide)
- Apply colors consistently: use accent colors sparingly for emphasis (titles, highlights, key data points)
- Ensure visual hierarchy through color contrast - primary content should stand out from secondary elements
- Maintain sufficient white space - avoid overwhelming slides with too many competing colors
- Use at most 2-3 colors from the palette on this slide for consistency

## Layout Strategy
- Unified full-page layouts without fragmentation
- Present only the most important information concisely
- Use typography and spacing to guide reader attention
- Maintain consistent color scheme and visual elements throughout
- Containers should align with each other
- Use grid layouts in containers to fill all of the page in horizontal direction
- Use justify-evenly to fill containers with elements
</design_guidelines>

<prohibited_practices>
- Don't overflow under the requirement of min-height: 720px (target height)
- Don't use vertical layout when using images or charts. Don't put chart/image vertically stacked
- Never write any CSS code in body tag, never use overflow: hidden
- Never use `position: absolute` for main containers (breaks responsiveness)
- Never reference non-existent or non-local image paths
</prohibited_practices>

<visualization_requirements>
- Remember that Chart.js is version 3; do **not** use version 2 syntax such as horizontalBar
- DO NOT include radar charts; they typically result in poor visual appeal
- When using Chart.js, ALWAYS wrap `<canvas>` elements in a parent div with explicit pixel height: `<div style="height: 300px;"><canvas id="myChart"></canvas></div>`
- Each column may contain **maximum one** chart/graph/image
- For data visualization, use Chart.js or D3.js instead of searching images and give the source of the data
- Only write charts when there's solid data collected from the data source
- For charts/visualizations, use harmonious color progressions derived from the palette colors
- Always provide verifiable data sources; never generate fictional numerical data
- Use Chart.js for standard charts, D3.js for complex custom visualizations
</visualization_requirements>
